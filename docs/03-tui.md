# 03 · TUI 规格（human）

> **实现边界**：`human` 的 completion TUI 已实现队列/接单拒单、文本/进度/工具事件、任务级命令开关、镜像准备与按 `R` 扫描的交付 review/preview/确认、WebSocket 重连与幂等 outbox 等最小闭环。当前没有 fsnotify 后台 watcher。本文也保留图片、完整 Audit、onboarding、todo/通知、IDE 快捷入口等产品规格；未在当前代码落地的体验项不能仅凭本篇视为已交付。

## 1. 定位：沟通与交付的信使台

TUI **不是**逐回合扮演模型的驾驶舱——真正的研发在**接单人自己的 IDE** 里发生（Claude Code、cursor、手写，随意）。TUI 只管两件事：

- **沟通**：读懂对方要什么、聊清楚（对话形态）；
- **交付**：活在 IDE 里干完了，回 TUI 核对改动、放行给对方，附一句说明。

中间几十上百个 tool-call 回合**不暴露给人**——它们在接单人的 IDE 里发生。人面对的是**任务级**节奏（聊需求 → 去干活 → 回来交付），不是 tool-call 级。这跟真人外包的工作方式一致，也一举消解了"人肉逐回合应答会累死"的人体工学问题。

**文件是隐式的**：接单人在镜像工作目录里改（用任何工具），回 TUI 后按 `R` 扫描改动并汇成待核对 diff，再放行（§5）。人不手搓任何 tool_call——那是底层管道。

三原则：**读懂需求**（对话渲染要清晰）、**在自己地盘干活**（TUI 不抢 IDE 的活）、**交付前核对**（所见即所发）。

## 2. 沟通形态：对话，不能主动发起

需求方那头是 harness，它以为在跟一个模型对话——**只能它发起，人只能响应**（completion 协议无反向通道，codex/claude 均不支持模型主动找用户）。所以：

- **就是对话**：对方在它的 harness 里说需求 → 作为请求进来 → 人在 TUI 读到、回复。人的回复可以是：**交付**（说明文字 + 工作目录改动）、**纯澄清反问**（"确认下：登录页还是注册页？"——对 harness 就是模型的一次澄清回复，显示给用户，用户答 → 新请求进来）、或给中间说明。多轮澄清天然就是多轮对话。
- **进度可流式，不必憋到干完**：实现允许请求靠心跳保活，并让人在 SSE 流里先吐进度（"在复现了""大概还要半小时"），最后再给交付——把长任务黑箱变成"看得到进度"。但真实外部 harness 的 10m/2h 长挂尚未执行，数小时能力目前是待验证目标，不是产品承诺。
- **想说的话只能搭在响应里（"tips 过去"）**：没有 pending 请求时人无法主动找对方（无通道）。所有主动信息（提醒、建议、"你也看下 X"）都搭在下一次响应里带出去。独立 delegation 模式已有 A2A task/message 通道，但当前实现未声明 push notification/webhook。
- **转包退休为常态**：接单人用自己的 Claude/Codex 在工作目录里干活，本就是默认，TUI 不关心你用什么——"转包"不再是需要特殊设计的功能，就是"人用他的工具干活"。人对交付负责（核对闸），数据出境需披露（[02](02-gateway.md) §11）。

## 4. 上下文渲染（读懂需求的关键）

- **system 提示**默认折叠（往往很长且每回合重复），可展开；
- **消息流** markdown 渲染；人自己的进度/交付/反问也在流里，形成连续对话感；
- **工具结果**（对方 harness 在其循环里产生的、需要人看的部分，如报错、日志）语法高亮 + 折叠 + 懒加载；
- **会话续接**：只高亮本请求新增的尾部，人不必重读历史；
- **路径虚拟对齐**：对方路径已从真实根改写为 `/workspace`（隐去 home/用户名，[02](02-gateway.md) §11）；镜像目录按同一坐标系落盘；
- 图片内联（对方消息里的截图，kitty/iTerm2，降级占位）。

## 5. 工作区同步与镜像（文件的隐式交付通道）

**镜像目录**：对方工作区在接单人机器上的本地副本（`~/mirror/<caller_id>/<workspace_key>`，坐标系 `/workspace`；按正确性命名空间而非会话建目录，[05](05-p1-m0-contract.md) §1）。**研发在这里发生，用接单人自己的 IDE/agent；TUI 不插手过程，当前在按 `R` 时扫描改动。**

**铁律：唯一真相是对方的工作树，镜像只是草稿板、不是第二份真相。** 镜像 = "接单人读到/知道的东西"物化成文件（等价于 Claude 上下文里的文件视图，只是落了盘），从真相播种、随时可丢、有疑问就重新同步。交付的是"施加在对方树上的一条条编辑"，**绝不是"我这份镜像"**——严禁把整个镜像当第二份真相回传。两边靠**逐 tool-call 对账**保持一致（不是"一致性消失"，是缩小到每次 edit 的确认）：baseline 只在收到**成功 tool result** 后推进；对方手改 / formatter / 多文件部分失败都表现为 edit 响亮失败 → 重新 read 回填重做（fail-explicit）。**唯一例外是 `write` 整文件覆盖**（无 `old_string` 保护，会静默盖掉对方并发改动）——故已有文件优先 `edit`，`write` 只用于新文件或明确整体替换。

### 5.1 git 只用于"上下文"，不用于"传输"

- **git-for-context**：给接单人一份可导航、基线正确的完整代码库（人和其 agent 能 grep、读结构）。**这是接单人侧需要 git 的唯一理由。**
- **git-for-transport**：**不存在**。改动回传永远走 tool_call 通道（edit/write 由对方 harness 执行），绝不走 git push/pull——否则同一改动被 git 与 tool_call 双重应用，必然打架。

git 只在**建立时用一次**给接单人正确起点；此后两边 git 各过各的，只有文件增量单向（镜像→对方）经 tool_call 流动。

### 5.2 回传认内容不认分支：两边不同分支不会导致 apply 冲突

- **传输是 edit/write tool_call，匹配文件内容、不认分支**：`edit(path, old_string, new_string)` 成功的唯一条件是对方**当前**文件里找得到 `old_string`——分支名、commit hash 一概不影响（比二期 `git apply` 更宽容：无需 base blob）。接单人就像"坐在对方键盘前的模型"，改动落在对方**当前 checkout 的树**上；接单人本地的 `human/<task_id>` 分支只是私有草稿纸，**从不作为 git 被 apply**。故不存在"两边分支不同 → apply 冲突"。
- **唯一真实风险是内容基线不同（非分支不同）**：镜像播种内容 ≠ 对方此刻内容（如 clone 了 C0 但对方工作树有未提交改动）→ 对应 seam 的 edit 失败——**响亮报错、只伤一处、自愈**（重新 read 回填 → 重做），详见 5.4。正解不是"两边同分支"，是"镜像从对方真实当前内容播种"（对方从干净已 push 基线开工最稳）。

### 5.3 三档镜像（按任务）

| 档 | 接单人侧 | 适用 | 基线 |
|---|---|---|---|
| 惰性 scratch（无 git） | 只落被读到的文件 | adb 调试、排障、小改 | 无需 |
| 完整 clone（共享 remote） | `git clone` + checkout 基线 commit | 研发类 | 对方 HEAD 已 push |
| 完整 bundle（无共享 remote） | 网关中转 repo 打包 | 研发但无 repo 权限 | 打包 commit |

### 5.4 交付、漂移与兜底

- **交付**：当前按 `R` 扫描镜像，核对面板列出自上次确认基线起的增/改/删；人 preview 并确认后，系统打包成文件 tool_call 随本次响应发出。**默认核对后发**（所见即所发 + 安全出口）。fsnotify 后台自动刷新仍是体验 backlog。
- **漂移源**：对方人手动改文件、对方侧 formatter/hook 改格式、基线脏——都表现为 **edit tool_call 失败**（old_string 匹配不上），自带信号；兜底是重新 `read` 对方该文件真实内容回填镜像、人重做那一处（即二期 F3 冲突处理降到 tool-call 粒度）。最佳实践：对方从**干净、已 push** 的基线开工。
- **能力降级**：对方 harness 无文件写入工具（只读研究型 agent）→ 交付通道不可用，退回纯对话，状态栏 `文件✗`。

## 6. 键位（默认）

| 键 | 作用域 | 行为 |
|---|---|---|
| `1-3` / `Tab` | 全局 | 切换视图 |
| `a` / `r` | Queue | 接单 / 拒单 |
| `c` | Desk | 聊天/进度输入（回复、澄清反问、流式进度） |
| `x` | Desk | 对方机器命令（环境绑定型任务用，如 adb；研发本地命令在你自己终端） |
| `R` | Desk | 交付核对面板（逐项确认增/改/删） |
| `e` / `o` | Desk | 在镜像目录打开 $IDE / 新终端（去干活或起本机 agent） |
| `m` | Desk | 待办（打勾/新建/排序；可选输出进度给对方） |
| `space` | Desk | 展开/折叠光标处折叠块 |
| `Ctrl+P` | Desk | 预览交付（本次响应：说明文字 + 文件改动 + 命令） |
| `Enter` | 预览 | 确认发送 |
| `?` | 全局 | 键位帮助 |

## 7. 关键流程

**S1 环境绑定型（adb 调试，回合短、人肉在场）**：接单 → 读诉求 → `x` 发 `adb logcat -d`（对方机器执行、结果下一回合回）→ 读日志定位 → 改镜像里的文件 → `R` 核对 → `c` 说明 → 交付 → 循环。人是这台设备的远程大脑。命令密集、文件少。

**S2 研发型（目标流程，长任务、agent 驱动、人监督）**：接单 → `c` 与对方聊清需求（有歧义就反问）→ 在自己的 IDE 用 Claude Code 处理镜像 → 期间可 `c` 吐进度 → 干完回 TUI，按 `R` 扫描并核对 diff → `Ctrl+P` 预览 → 交付。当前未实现 `e` 键打开 IDE 与后台 watcher；真实长挂能力仍受 P1-M0 外部门约束。

**发送预览安全高亮**（[02](02-gateway.md) §11）：预览对**越界路径**（网关拒转，红色阻断）、**敏感路径**（`.env`/`.ssh`/密钥，黄色）、**危险 shell**（`rm -rf`/`curl|sh`/`sudo`，黄色）显式高亮。

**拒单/离线**：准入时无人在线 → 200 前返回 overloaded；已推送后按 `r` 拒单或等到 `max_pending` → 阶段 B 流内 error / 断流。两条路径已有仓库内测试；真实客户端对部分响应的重试行为仍待 P1-M0 外部实测。

**中断**：对方 Esc/断连 → 横幅"对方已取消本请求"，人停手。

**恢复**：TUI 崩溃重连 → 挂起请求仍在 gateway 队列（心跳中）→ 对话草稿与镜像目录都在本地 → 续答。

## 8. 配置

经 `viper` 加载：配置文件（toml）+ 环境变量 + flag 层叠；CLI 由 `cobra` 提供子命令。

```toml
[gateway]
url   = "wss://humand.example.com/internal/v1/worker/ws"
token = "hae_..."

[workspace]
mirror_root = "~/mirror"

[worker]
outbox = "~/.human/worker-outbox.db"
```

以上四个键是当前 completion TUI 实际读取的配置；身份显示名、IDE 命令、通知和图片协议仍是产品规格，不应提前写入可运行样例。
